기초 · 왜 중요한가

컨텍스트 윈도란? 왜 아껴야 하나

  • 컨텍스트 윈도 = 모델이 한 번에 "보는" 전체 토큰 — 대화·읽은 파일·명령 출력 전부가 쌓임
  • "가장 중요하게 관리해야 할 자원" — 디버깅/탐색 한 세션이 수만 토큰을 순식간에 소비
  • 찰수록 성능 저하(context rot): Claude가 초기 지시를 "잊고" 실수↑
책상에 비유: 자료를 무한정 쌓을 순 없다. 원시 데이터로 책상을 덮으면 정작 생각할 공간이 사라진다.

그래서 → ① 스킬은 필요할 때만 펼치고(progressive disclosure) ② 원시 출력은 샌드박스로(context-mode) ③ status line으로 사용량을 본다(뒤에서).

출처 code.claude.com/docs/en/best-practices(1차): "context window fills up fast, performance degrades as it fills … the most important resource to manage" · Part 2에서 n²·attention budget 심화

superpowers

주인공 ② context-mode

"Think in Code" — 답만 컨텍스트에 넣어라

  • 문제: 모든 MCP 도구가 원시 데이터를 컨텍스트에 쏟음. 정의(입력)+결과(출력) 양쪽에서 소모 → 30분 만에 40% 증발
  • 해법: 분석을 계산하지 말고 프로그래밍하라. 700KB 로그를 읽으면 700KB의 추론 능력을 잃음 → 샌드박스에서 돌리고 console.log로 답만 회수
영감: Cloudflare Code Mode는 도구 정의(입력)를 압축 → context-mode는 도구 출력을 압축. "같은 원리, 반대 방향".
// Before: 47 × Read = 700 KB
// After:  1 × ctx_execute = 3.6 KB
ctx_execute("js", `
  const f = fs.readdirSync('src')
    .filter(x => x.endsWith('.ts'));
  f.forEach(x => console.log(
    x + ': ' + lines(x)));
`)

벤치마크(저자 보고*): 315KB → 5.4KB (98%), 세션 ~30분 → ~3시간

출처 로컬 README·CLAUDE.md·ctx_execute 도구설명(verbatim) · mksg.lu/blog/context-mode(저자) · blog.cloudflare.com/code-mode

* 98%·6x·HN점수·사용자수는 저자 자체 보고 — "저자에 따르면"으로 프레이밍, 독립 검증 아님. 이 세션도 context-mode가 돌고 있어 WebFetch가 ctx_fetch_and_index로 강제 리다이렉트됨 → 라이브 데모 가능.

주인공 ② context-mode

어떻게 작동하나 — 11개 도구 + hook 강제

  • 6 sandbox: ctx_execute(_file) · ctx_batch_execute · ctx_index · ctx_search · ctx_fetch_and_index
  • 5 meta: ctx_stats · doctor · upgrade · purge · insight
  • 격리 subprocess(12개 언어)에서 stdout만 회수 · FTS5/BM25 지식베이스(SQLite)
  • Session Continuity: compaction 시 작업상태(파일·태스크·결정)를 복원
핵심: SKILL.md는 "안내할 뿐 막지 못함"(~60%) → PreToolUse/PostToolUse/PreCompact/SessionStart hook이 차단·리다이렉트해 ~98% 강제.

정의

"하네스(harness)"란?

모델을 둘러싸고 일을 시키는 레이어. Anthropic은 Claude Agent SDK(구 Claude Code SDK)를 "general-purpose agent harness"라고 직접 명명.

설계 원칙

"에이전트에게 컴퓨터를 줘라" — 터미널/bash, 파일 read·write·search. 이 덕분에 Claude Code가 코딩 도구에서 범용 agent 플랫폼이 됨.

왜 필요한가 ①

컨텍스트는 유한 자원이다 — "context rot"

  • Context engineering = 프롬프트 엔지니어링의 후속. 추론 내내 (시스템 지침·도구·MCP·외부 데이터·대화 이력) 토큰을 큐레이션·유지하는 전략
  • 토큰이 늘수록 정확한 회상 능력이 떨어진다 (context rot) — 한계 효용 체감
  • 근본 원인: 트랜스포머의 n² pairwise attention + 유한한 "attention budget"
시사점: "컨텍스트 창을 더 키우면 된다"가 아니라 무엇을 넣고 뺄지 설계해야 한다.

출처 anthropic.com/engineering/effective-context-engineering-for-ai-agents · research.trychroma.com/context-rot (18-모델 연구) · n² = arXiv:1706.03762 (1차, 3-0)

왜 필요한가 ②

병목은 "더 큰 컨텍스트"가 아니라 하네스 설계

프런티어 모델(Opus 4.5)을 SDK 루프로 여러 컨텍스트 창에 걸쳐 돌려도, "claude.ai 클론 만들어줘" 같은 고수준 프롬프트만으론 프로덕션급 웹앱을 완성하지 못한다. compaction만으론 부족.

해법: 교대 근무하는 엔지니어 모델 (two-agent)

Initializer agent

첫 실행에서 환경을 세팅

Coding agent

매 세션 점진적 진척 + 다음 세션을 위한 명확한 산출물(git commit, progress 파일)을 남김

비유: "기억 없이 교대로 출근하는 엔지니어 팀" — 인수인계 아티팩트가 핵심.

개발자 관점 · 정직한 평가

장점 / 단점 — 개발자가 실제로 느끼는 것

장점

  • 긴 세션도 싸고 쾌적 — 메인 스레드 격리(iter당 ~2–5K 토큰, ~50K 아님) → 몇 시간 무인 실행
  • 잊어도 적용되는 거버넌스 — Task hook이 audit token 없는 보고를 exit 2로 반려 → "깨진 코드가 PR로 못 샌다"
  • 프로젝트당 1명령·기능당 1명령으로 리뷰·게이트된 PR까지
  • walk-away 한도 보장(--max-iter 50 / --max-cost $500 / 재개 상태)
  • diff에 따라 리뷰어 자동 확장 + QA 페르소나 · 아키텍처 결정은 패널로 사용자에게

단점 / 트레이드오프

  • source-passing ≠ runtime-working — 릴리스 게이트는 소스·CLI help만 검사. "1788 테스트"는 배선 증명이지 PR 품질 아님(예: 커밋 e28b320 "instrumentation dead on arrival"인데 suite는 green)
  • 설정·유지 복잡도↑(README ~790줄, 멀티플랫폼/테마 매트릭스, vendored lib drift)
  • 다중 에이전트 토큰 비용↑(scope+impl 이중 디스패치 ~15–20%) · thrift "~3x"는 목표치
  • 강한 강제는 Claude Code 한정(Cursor/Gemini는 soft) · --qa엔 dev 서버+Playwright 필요
  • 학습곡선·이름함정(테마 A–E·phase·3a/b/c·토큰) · agent-all엔 토대 preflight 없음

정리 & Q&A

핵심 5

오늘의 Takeaways

  1. Claude Code = 플랫폼. plugins·skills·MCP·subagents·hooks 5표면을 조합.
  2. 스킬은 "필요할 때 펼치는 매뉴얼", 하네스는 "공정 라인". 차이는 누가 계획을 쥐는가.
  3. 병목은 큰 컨텍스트가 아니라 하네스 설계. compaction·노트·subagent로 장기 작업 지탱.
  4. 추세는 프롬프트 → 오케스트레이션. /goal·workflows·OMO 모두 같은 방향.
  5. 우리 하네스의 차별점 = hook 강제 거버넌스 + 멀티플랫폼 포트.
실천: 내일 자기 프로젝트에서 /agent-init → 작은 기능 하나 /agent-all로.
